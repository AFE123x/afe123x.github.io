Chapter-1

chapter-1-Introduction

  • You can use qemu and the ESP32 to write RISC-V Assembly.

  • Obviously, the AMD and Intel Instruction sets are bloated, and extended for compatability reasons.

  • Assembly lets you exploit the hardware, more control than using C/C++. RISC-V Makes it easy, and it may become standard.

  • RISC-V is an attractive architecture, since it’s free and open source. The RISC-V Instruction set is clean compared to x86.

1.1: History of RISC:

  • The RISC (Reduced Instruction Set Computer) architecture began as a research project conducted by John L. Hennessy at Stanford between 1981 and 1984.
  • MIPS (Micrprocessor without Interlocked Pipeline Stages) was developed with the IBM 801 and the Berkeley RISC Project.
  • The objective of RISC was to be simpler than CISC of that time.
    • Simplicity is good, since chip manufacturers want to lower cost (fewer transistors), lower power requirements and it’s less complex for the software developers.

Chapter 4 - Computer Architecture

4.1: The Program Counter

  • This is the register which points to the address of the next instruction to perform.
    • It also handles conditional branching.
  • The standard RISC-V INstruction has a 32 bit word length, which needs to be aligned.
  • There may be extended instructions that are half word, or 16 bits.
  • After executing an instruction, the PC will increment by 2 or 4 bytes depending on the instruction.

4.3: Endianness

  • The RISC-V architecture is little endian and byte addressable.
    • This means lowest byte is addressed first.

4.4: General Purpose Registers

  • The register width is defined by a constant XLEN in an architecture.
    • if the system is 32 bits, like the ESP32, the XLEN = 32.
    • If the system is 64 bit, BLEN = 64.
  • There are typically 32 registers, from X0 - X31.

4.5: Subsets

  • The RISC-V breakns the architecture design into subsets with names like RV32I.
    • This lets manufacturers implement the sections needed, so, for example, manufacturers can choose what to include to reduce complexity.
  • The most basic architectural subset is RV32I. This architecture:
    • Defines the most basic opcodes
      • Register set and memory operations as a base architecture.
    • 32 defines that it’s a 32 bit system (32 bit wide registers).
  • RV64I is the base 64 bit architecture
  • RV32E is the 32 bit system for embedded processors.
  • RV128I is a 128 bit system.

4.6 - Register Specifics

  • x0 register: This register always has 0.
    • When the source register is x0, it puts 0 is the dest register.
    • When the destination register is x0, it discards a value.
  • flag bits: RISC-V doesn’t have flag bits, since it adds complexity.
    • Instead, some instructions may store a flag bit in the destination register.
  • x1 register: This register is the return address register.
    • This is by the GNU convention, and not legit. when a function is called, the return address is stored in the ra registers.
      • There are opcodes that arrange this.
  • x2 register: This register is your Stack Pointer.
    • the program will typically add/subtract from the pointer.
  • x3 / gp register: This is the GNU convention for RISC-V, which serves as a global pointer.
  • x4 / tp: This register, as per the GNU convention, is used for thread local storage access.
  • x5-x7 / t0-t2: These are temporary registers, which are registers the caller is responseble for saving the values.
  • x8 /s0 / fp: The x8 register serves two purposes: s0 is the first saved register value, and the fp is the stack frame pointer. This register must be saved and restored by the callee.
  • x9 / s1: This is the second saved register value. callee is responsible for saving and restoring register.
  • registers x10-x11 / a0-a1: These are argument registers. these are 32 bit registers on RV32. If you want to move a 64 bit type, you insert the least significant 32 bits in a0, and the most significant 32 bits in a1 (endianness convention).
  • registers x12-x17 / a2 - a7: We pass arguments into registers from a2 to a7.
  • registers x17-x27 / s2 - s11: These are callee saved registers.
    • The callee needs to save and restore the registers.
  • registers x28-x31 / t3 - t6: This is the caller save registers.

Chapter 5.2: Simple Exercise

#include<stdio.h>
    
    int add3(int arg1, int arg2, int arg3){
        arg1 = arg1 + arg2 + arg3;
        return arg1;
    }
    
    int main(int argc, char** argv){
        int a = 23, b = 24, c = 25;
        int r = 0;
    
        r = add3(a,b,c);
        printf("r = %d\n",r);
        return 0;
    }

When we compile this, we get the following assembly:

    .globl  add3 # Tells the linker that add3 exists. 
        .type   add3, @function
    add3:
        addw    a0,a0,a1 # arg0 = arg0 + arg1
        addw    a0,a0,a2 # arg0 = arg0 + arg2

The format of the assembly looks like:

Field1  Field2   Field3      Field4
    label:  opcode  operands    comments

Pseudo Opcodes

  • Pseudo opcodes influnce the workings of the assembler.

  • global symbol: A global symbol tells the linker that, for example, add3 exists.

.global add3, printf
  • This tells the linker that add3 and printf exists.
    • It’s good practice to list all external symbols. By default, symbols are global.
  • .text symbol: This pseudo opcode indicates that the code belongs under an object code section known as “.text”
    • This tells the linker where to store it in memory.

5.3: add opcode

add3:   add a0,a0,a1
  • This will perform a0 + a1, and store it in a0.

5.4: objdump

  • We can do objdump to inspect the binary.
riscv64-unknown-elf-objdump -d hello.o

Chapter 6 - Load and Store Memory

6.1: word sizes

  • On RISC-V, here are the word sizes:
    • byte (8 bits)
    • half word(2 bytes)
    • word (4 bytes)
    • double word (8 bytes)

6.2: Load Instructions

  • when we perform the load instruction, we need to specify the following:
    • Destination register name(rd)
    • The address (the assembler symbol)
    • the word (unit) size
lb  rd, symbol #load signed byte
    lh  rd, symbol #load signed half word
    lw  rd, symbol #load signed word
    ld  rd, symbol #load signed double word: RV64 only

6.3: Sign Extension

  • When you add a 32 bit signed value, and you want it to be a 64 bit value, the hardware will do a sign extend, where it extends the MSB.

6.4: the .data section

  • This stores data values in a section of memory belonging to data. similar to establishing static variables in C and C++.

6.5: unsigned values

  • if we don’t want to perform a sign extension, we can use the unsigned load instructions.
lbu rd, symbol # Load unsigned byte
    lhu rd, symbol # Load unsigned half word
    lwu rd, symbol # Load unsigned word: RV64I only
    ldu rd, symbol # Load unsigned double word: RV128I only

6.6: Alignment

  • Alignment is done to optimize performance:
.balign n, [fill], [max]
  • n tells us how many bbytes to align.
  • fill is what we fill the padding with (it’s zero by default if not filled).
  • max is the maximum number of bytes to pad with when specified.

6.7: Immediate Values

  • It’s inefficient to read constants from memory, so why not read it from the instruction?
    • This is known as immediate data.
lb t1, one
    add a1,t1,zero # a1= t1 + 0
    
        .section .srodata
    one .byte   1

6.8: the li pseudo op

  • The use of immediate data eliminates the need for defining the data type for the load instruction, we can use li.
li  t1,1
    add a1,t1,zero

6.9: the addi opcode

  • This is similar to load, but with the immediate value.
addi a1, a1, 1 #a1 = a1 + 1

6.10: the mv pseudo op

  • We may need to copy/move data from one register to another. There’s two ways to do it:
add t0, t1, x0 # t0 = t1 + 0
    mv t0, t1
  • These two do the same thing.
